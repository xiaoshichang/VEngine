# macOS Packaging Design

## 1. Scope

The macOS packaging flow is driven from the Editor. It produces a runnable `.app` bundle for the currently opened
project while keeping `VEngineMacPlayer` itself as a normal executable target.

Current scope:

- Platform: macOS.
- Player binary: `VEngineMacPlayer`, built through the CMake/Xcode build tree before packaging completes.
- App bundle: generated by the Editor packer under the project `Build/Mac/` directory.
- Assets: runtime asset files referenced by the Editor asset database, including required builtin runtime assets.
- Asset metadata: exported `AssetManifest.json`.
- Managed scripting: app-local .NET JIT payload, the managed `VEngine.ScriptHost` files, and compiled project script DLLs.
- Build execution: the package modal advances one step per frame. The macOS flow explicitly runs CMake configure to
  update/export the Xcode project and then builds the `VEngineMacPlayer` target. Progress reaches 100% only after the
  player build, app assembly, signing, and package metadata steps complete.

## 2. Project Build Root

Packages are written under the opened project root:

```text
ProjectRoot/
  Assets/
  Library/
  Build/
    Logs/
    Mac/
```

`Build/` is generated local output and should not be required for source control.

## 3. Package Output Layout

Each packaging run creates a timestamped `.app` bundle:

```text
ProjectRoot/
  Build/
    Logs/
      Package_Mac_YYYYMMDD_HHMMSS.log
    Mac/
      ProjectName_YYYYMMDD_HHMMSS.app/
        Contents/
          Info.plist
          MacOS/
            VEngineMacPlayer
            *.dylib
          Frameworks/
            *.framework
          Data/
            VEProject.json
            AssetManifest.json
            Scripts/
              ProjectName.Scripts.dll
              ScriptAssembly.json
            Assets/
              ...
            BuiltinAsset/
              ...
            Library/
              Imported/
                ...
          Resources/
            PackageInfo.json
            Managed/
              VEngine.ScriptHost/
                VEngine.ScriptHost.dll
                VEngine.ScriptHost.runtimeconfig.json
                ...
            DotNet/
              osx-arm64/
                10.0.9/
                  dotnet
                  host/
                  shared/
          Logs/
```

Directory responsibilities:

- `Contents/MacOS/`: macOS executable payload for the packaged player.
- `Contents/MacOS/*.dylib` and `Contents/Frameworks/*.framework`: native dynamic libraries copied from the player build
  output directory when present.
- `Contents/Data/`: project data consumed by the runtime. The runtime `VEnginePlayer` package probe finds this directory
  as the sibling `Data/` directory under `Contents/`.
- `Contents/Data/Scripts/`: compiled project script DLLs and their script assembly manifest.
- `Contents/Resources/Managed/VEngine.ScriptHost/`: managed bridge used by the JIT scripting backend.
- `Contents/Resources/DotNet/osx-arm64/10.0.9/`: app-local .NET runtime and `hostfxr` payload.
- `Contents/Resources/PackageInfo.json`: package metadata for diagnostics and future tooling.
- `Build/Logs/Package_Mac_YYYYMMDD_HHMMSS.log`: per-run packaging log produced by the Editor packaging flow.

## 4. Packaging Steps

The Editor modal presents and executes these steps:

1. Prepare the `.app` bundle directories.
2. Refresh the asset database.
3. Export `AssetManifest.json`.
4. Copy runtime assets.
5. Copy compiled project managed scripts when present.
6. Run CMake configure for the macOS/Xcode build directory.
7. Build the `VEngineMacPlayer` target.
8. Copy `VEngineMacPlayer` into `Contents/MacOS/`.
9. Copy the macOS managed runtime payload into `Contents/Resources/`.
10. Write `Info.plist`.
11. Ad-hoc sign the generated `.app`.
12. Write `PackageInfo.json`.

If a step fails, packaging stops, the modal shows the failed step, and the packaging log records the failure code or
exception message.

## 5. Known Follow-Ups

- Add Debug/Release package configuration selection.
- Let users choose an output directory.
- Add explicit signing identity and entitlements controls.
- Add package cleanup/archiving options.
- Add a background worker for expensive package steps once the UI needs it.
